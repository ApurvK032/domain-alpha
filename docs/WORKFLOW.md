# Workflow Overview

## Purpose

This workflow turns raw domain ideas into a ranked shortlist for manual review. The screener should make the first pass systematic, explainable, and conservative so weak purchases are filtered out before any money is spent.

## Workflow at a Glance

1. Collect domain candidates.
2. Normalize the input data.
3. Enrich candidates with cost, demand, comparable-sales, and risk signals.
4. Score each domain with transparent weighted rules.
5. Apply cost and trademark-risk guardrails.
6. Rank candidates and assign `buy`, `watch`, or `avoid`.
7. Manually review the shortlist.
8. Track decisions and outcomes for future model improvement.

## Folder Mapping

- `docs/` - product, workflow, and idea documentation.
- `data/samples/` - small safe example files that can be committed.
- `data/raw/` - local raw exports from registrars, keyword tools, marketplaces, or trademark searches.
- `data/processed/` - local normalized and enriched datasets generated by scripts.
- `outputs/` - local ranked result files and shortlists generated by the screener.
- `scripts/` - one-off data prep, import, or analysis scripts.
- `src/domain_screener/` - reusable screener package code.
- `tests/` - scoring, validation, and workflow tests.

`data/raw/`, `data/processed/`, and `outputs/` are ignored by default except for placeholder files. Keep private exports, paid datasets, and generated result files out of git unless they are intentionally sanitized samples.

## MVP Batch Process

### 1. Candidate Intake

Start with a CSV of domain candidates. Each row should represent one domain and include known costs, demand estimates, comparable-sales notes, brandability ratings, and trademark-risk notes.

### 2. Validation

The screener should confirm required columns exist, numeric fields are parseable, and action-critical fields are not missing. Invalid rows should be reported clearly rather than silently skipped.

### 3. Data Enrichment

Early enrichment can be manual or semi-automated. Later versions can add registrar availability checks, keyword-volume tools, trend data, sales-history datasets, and trademark search APIs.

### 4. Scoring

The MVP scoring model should remain deterministic and explainable:

```text
domain_score =
  demand_score
+ comparable_sales_score
+ brandability_score
+ tld_score
- cost_penalty
- trademark_risk_penalty
- renewal_cost_penalty
```

Each component should be visible in the output so a user can audit the final ranking.

### 5. Guardrails

Some risks should override a good-looking score. A domain should usually be downgraded to `avoid` when it has high trademark risk, unusually high renewal cost, unclear ownership or availability, or a resale thesis based only on hype.

### 6. Ranking and Review

The screener should export a ranked file and highlight the top 20-50 candidates. The user should manually review those candidates before purchase, including trademark checks, comparable-sale quality, renewal pricing, and marketplace fit.

### 7. Outcome Tracking

When domains are bought, watched, sold, dropped, or renewed, those outcomes should be recorded. This creates the dataset needed for better thresholds and future supervised learning.

## Draft Action Labels

- `buy` - strong score, acceptable cost, clear demand thesis, and no obvious legal risk.
- `watch` - promising but incomplete, expensive, volatile, or dependent on more evidence.
- `avoid` - weak score, poor resale thesis, high cost, high renewal burden, or meaningful trademark risk.

## Hygiene Rules

- Commit only safe samples and reusable source code.
- Keep raw third-party exports and generated results out of git.
- Prefer structured CSV, JSON, or SQL tables over free-form notes for screener inputs.
- Keep scoring rules configurable and easy to inspect.
- Treat trademark and financial conclusions as decision-support signals, not advice.
